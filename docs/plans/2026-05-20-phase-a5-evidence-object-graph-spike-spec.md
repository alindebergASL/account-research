## Schema contract

Use Zod. Keep enums explicit and narrow enough to validate, but not so narrow that the spike cannot express useful account facts.

### ID conventions

IDs should be deterministic for spike fixtures where practical:

- `srcdoc_<slug>`
- `ex_<sourceSlug>_<ordinal>`
- `claim_<slug>`
- `obj_<type>_<slug>`
- `edge_<slug>`

The validator should not require a specific prefix, but tests should use readable IDs.

### EvidenceExcerpt

```ts
export const EvidenceExcerpt = z.object({
  id: z.string().min(1),
  source_document_id: z.string().min(1),
  text: z.string().min(20),
  char_start: z.number().int().nonnegative(),
  char_end: z.number().int().positive(),
  extraction_method: z.enum(["exact_span", "normalized_span", "model_suggested_verified", "manual"]),
  captured_at: z.string().datetime(),
  metadata: z.record(z.string(), z.unknown()).default({}),
});
```

Validator rules:

- `source_document_id` must exist.
- `char_start < char_end`.
- `source.content_text.slice(char_start, char_end)` must exactly equal `text` for `exact_span`.
- For `normalized_span`, normalized whitespace text must match normalized source slice.
- Excerpts must not be invented by the model. They must map to source text.

## Failure-and-recovery cascade rules

A.5 does not need UI, but validator/report should model cascade impact.

Define a pure function:

```ts
export function computeCascadeImpact(graph: AccountGraphDocument, event: GraphCorrectionEvent): CascadeImpact
```

Minimum event types:

- `claim_marked_wrong`
- `source_marked_unreliable`
- `evidence_excerpt_invalidated`
- `account_object_marked_wrong`

Minimum cascade rules:

1. Claim marked wrong:
   - Claim status becomes `marked_wrong`.
   - AccountObjects referencing only that claim are downgraded to `marked_wrong` or `proposed_review_needed` in the impact report.
   - AccountObjects with multiple claims are downgraded in confidence and flagged for review.
   - Claim-to-claim derived/supports edges are flagged.

2. Source marked unreliable:
   - All excerpts from source are impacted.
   - All ClaimEvidence using those excerpts is impacted.
   - Claims supported only by impacted evidence are downgraded to low/unknown and review-needed.
   - Conflicts involving those claims should be recomputed/reviewed.

3. Evidence excerpt invalidated:
   - Related ClaimEvidence becomes invalid.
   - Related claims are downgraded or marked review-needed.

4. AccountObject marked wrong:
   - Its direct claims are not automatically wrong unless the object is merely a wrapper around one claim.
   - Related recommendations/actions are flagged review-needed.

A.5 can return cascade impact without mutating graph state. Mutation belongs later.

## Staged pipeline shape

A.5 must not ask the model to invent the graph from raw web context in a single pass.

Pipeline stages:

1. Source fixture/discovery input.
2. Source normalization and allowlist classification.
3. Excerpt extraction.
4. Claim/object assembly referencing evidence excerpt IDs.
5. Graph validation.
6. Optional repair attempt.
7. Report generation.

### Spike A: controlled-source graph generation

Purpose:
Measure whether Hermes/model-assisted graph assembly can create useful AccountObjects and Claims from a controlled source set while referencing only valid evidence IDs.

Input:

- Controlled Nueva fixture sources under `web/lib/accountGraph/fixtures/nueva/sources.ts`.
- Use 4-8 source documents. Rationale: enough source diversity to test cross-source evidence, contradiction, and provenance without turning A.5 into broad web research. If fewer than 4 sources cover all qualitative categories, document why. If more than 8 are needed, shrink the fixture or split it; source count is a scoping guardrail, not a pass/fail quality metric.
- Sources should include enough content to evaluate qualitative coverage across:
  - account snapshot
  - signal/change
  - initiative
  - stakeholder/persona hypothesis when the fixture evidence supports one
  - risk or open question
  - opportunity or recommended action

Output:

- `AccountGraphDocument` JSON.
- Validation report.
- Markdown result report.

Pass criteria:

- Zod parse success: 100%.
- Referential integrity: 100% of referenced source/excerpt/claim/object IDs exist.
- Evidence grounding: 100% of ClaimEvidence links reference existing excerpts; >= 95% of those linked excerpts pass exact/normalized text validation.
- Exact/normalized excerpt validity: >= 95% valid across accepted excerpts.
- No invented source IDs: 0.
- No invented excerpt IDs: 0.
- Qualitative coverage, not raw volume: the graph must represent each fixture-supported category that exists in the controlled Nueva source set, including account snapshot, signal/change, initiative, stakeholder/persona if supported, risk or open question, and opportunity or recommended action.
- Grounding coverage: every fact/signal/risk/opportunity/stakeholder/initiative claim intended for Brief or Workshop display must have at least one supporting or contradicting ClaimEvidence row, except explicitly marked open questions or unverified hypotheses.
- At least one claim/object uses `provenance_status="verified"` and none falsely claim verified without valid evidence.
- Validator rejects deliberately broken fixture cases in tests.

Threshold interpretation:

- 100% requirements are integrity invariants, not quality targets. Missing references, invented IDs, parse failure, or false verified provenance mean the foundation is unsafe. A miss by any margin is fail/repair, not “close enough.”
- 95% excerpt-validity thresholds allow for one-off fixture/extractor edge cases during spike work, while still proving deterministic grounding is mostly reliable. If a run lands between 90% and 95%, classify as borderline: do not proceed to A.6 automatically; inspect failures, fix deterministic extraction if possible, and rerun once. Below 90% is fail.
- The removed numeric volume requirements (previously “8 claims / 5 objects”) were qualitative coverage requirements dressed as numbers. Use fixture-supported coverage instead: the graph must express all important source-backed account intelligence categories present in the controlled source set.
- Grounding coverage is strict for displayable factual claims because the product thesis is evidence-first. If a claim cannot be grounded, it must be downgraded to unverified hypothesis/open question or excluded from ratified/displayable output.

Fail criteria:

- Any model output includes source/excerpt IDs not in supplied context.
- More than 5% excerpt mappings fail.
- Any high-confidence claim lacks strong/medium evidence.
- The graph cannot express a contradiction or uncertainty present in fixture data.
- Report cannot explain failures in reviewer-readable terms.

### Spike B: excerpt extraction reliability

Purpose:
Measure whether evidence excerpts can be extracted and verified deterministically enough to support durable claims.

Input:

- Same Nueva controlled sources.
- A table of expected quote-like snippets and paraphrase targets.

Output:

- Extracted `EvidenceExcerpt[]`.
- Matching metrics.
- Failure examples.

Pass criteria:

- Exact span extraction succeeds for >= 90% of expected direct quotes.
- Normalized span extraction succeeds for >= 95% of expected whitespace-normalized snippets.
- The extractor rejects paraphrases that do not appear in the source text unless explicitly classified as not extractable.
- Each accepted excerpt has correct `source_document_id`, `char_start`, `char_end`, and text slice.
- No excerpt shorter than 20 characters unless test explicitly documents why.

Threshold interpretation:

- The 90% exact-span threshold acknowledges that controlled fixtures may include punctuation, Unicode, whitespace, or boilerplate edge cases where exact copying is brittle. If exact span lands between 85% and 90% but normalized span is >= 95%, classify as borderline and inspect failures before repeating once. Below 85% is fail.
- The 95% normalized threshold is the real reliability bar for durable evidence capture. If normalized span lands between 90% and 95%, classify as borderline and fix deterministic matching before A.6. Below 90% is fail.
- Paraphrase rejection is a hard invariant: accepting paraphrases as excerpts defeats durable evidence. Any accepted paraphrase that is not present in source text is fail.
- Offset correctness is a hard invariant for accepted excerpts. A small miss is not acceptable because downstream audit links depend on exact source spans.

Fail criteria:

- Extractor frequently accepts paraphrases as excerpts.
- Span offsets are unstable or incorrect.
- Excerpt extraction depends on model trust instead of deterministic verification.

## Prompt contracts

Create prompt builders in `web/lib/accountGraph/prompts.ts`. Even if A.5 uses fake/stub model output for tests, the prompt contract must be explicit.

### Excerpt extraction prompt contract

The model, if used, may suggest candidate excerpts but must not be trusted.

Rules:

- Input includes source IDs and source text.
- Output candidate excerpts must include exact copied text from the source.
- Output must not paraphrase.
- Output must not invent source IDs.
- Deterministic validator verifies every candidate.
- Invalid candidates are rejected.

### Claim/object assembly prompt contract

The model receives:

- Account context.
- SourceDocument metadata.
- Valid EvidenceExcerpt IDs and excerpt text.
- Allowed enum values.
- Output JSON schema.

Rules:

- Claims may cite only supplied evidence excerpt IDs.
- Claims may not cite raw URLs directly; they cite EvidenceExcerpt IDs via ClaimEvidence.
- Claims with no evidence must use `provenance_status="unverified"` and confidence `low` or `unknown`, unless they are open questions.
- High confidence requires at least one `strong` ClaimEvidence link with role `supports`.
- Contradictions must produce `ClaimEvidence(role="contradicts")` or `GraphEdge(kind="contradicts")` plus optional `ConflictRecord`.
- Do not output Brief JSON.
- Do not output CanvasDocument.
- Do not produce UI components.

## Validator rejection logic

Create `validateAccountGraph(graph): AccountGraphValidationReport`.

Report should include:

```ts
type AccountGraphValidationReport = {
  ok: boolean;
  errors: ValidationIssue[];
  warnings: ValidationIssue[];
  metrics: {
    source_count: number;
    excerpt_count: number;
    claim_count: number;
    account_object_count: number;
    claim_evidence_count: number;
    valid_excerpt_ratio: number;
    claims_with_evidence_ratio: number;
    high_confidence_claims_without_strong_evidence: number;
    invented_reference_count: number;
    contradiction_count: number;
  };
};
```

Hard errors:

- Zod parse failure.
- Duplicate IDs within same collection.
- ClaimEvidence references missing Claim.
- ClaimEvidence references missing EvidenceExcerpt.
- EvidenceExcerpt references missing SourceDocument.
- Excerpt span does not match source text under its extraction method.
- GraphEdge references missing nodes.
- Claim or AccountObject references unknown account ID/name inconsistent with graph root unless explicitly scoped hierarchy reference.
- High confidence claim without strong/medium evidence.
- `provenance_status="verified"` without valid evidence.
- SourceDocument `allowed=false` used by verified claim.

Warnings:

- Medium confidence claim with weak evidence only.
- Object with zero claim_ids.
- Claim with no evidence and not open_question/recommendation.
- Stale/unknown freshness.
- PII risk medium/high.
- Source retention `do_not_store` with content_text present.

Repair policy:

- A.5 may attempt exactly one repair pass for model-created graph assembly errors.
- The repair prompt receives only validation issues and allowed IDs; it must not receive permission to create new source/excerpt IDs.
- If repair fails, reject and surface failure in report.
- Do not silently downgrade hard errors into warnings.

## Cost ceiling and run controls

A.5 should be cheap and mostly deterministic.

Default iteration budget:

- Fixture mode: $0 model/API spend. This is the default for implementation, local tests, CI-like verification, and PR review.
- Model mode: final validation only, explicitly invoked, never automatic. Target ceiling: $10 total for one Spike A + Spike B validation run. Hard ceiling: $25 unless Andrew explicitly approves a higher spend.

Script requirements:

- `--mode fixture` must never call external model APIs or web fetch APIs.
- `--mode model` must print estimated/planned model mode before running if possible and must record observed token/cost metadata when available.
- Add `--max-cost-usd` flag for model mode, default `10`. If estimated/observed cost exceeds the ceiling, stop additional model calls, mark the run `budget_exceeded`, preserve partial artifacts, and write the report.
- If exact provider cost is not available, track input/output tokens and call count, and mark cost as `unknown_estimated`; unknown cost cannot be treated as a pass for model-mode final validation.
- A budget-exceeded run is not a product failure. It is an operational failure requiring smaller fixtures, cheaper model, tighter prompts, or explicit approval before rerun.


If the project test command differs, use existing repo convention but document exact command in report.

Script modes:

- `--mode fixture`: deterministic no-network/no-model replay using fixture graph assembly. This is the iteration default and is required for CI/local reliability.
- `--mode model`: optional real model call if existing env supports it. Must be off by default and used only for final validation after fixture mode is passing, not for normal iteration.

For A.5, fixture mode is mandatory and should be the default path for development, tests, and PR verification. Real/model mode is final validation only and may be skipped if credentials or budget are not explicitly approved.

## Testing requirements

### `tests/accountGraph.schema.test.ts`

Test:

- Valid minimal AccountGraphDocument parses.
- Invalid enum values fail.
- Missing source/excerpt/claim required fields fail.
- Account hierarchy reference parses.
- MEDDPICC metadata parses.
- ConflictRecord parses.

### `tests/accountGraph.excerpts.test.ts`

Test:

- Exact span valid.
- Exact span invalid when offset wrong.
- Normalized whitespace match valid.
- Paraphrase rejected as excerpt.
- Missing source rejected.

### `tests/accountGraph.validation.test.ts`

Test:

- Valid graph passes.
- Duplicate IDs fail.
- Invented source/excerpt/claim references fail.
- Verified claim without evidence fails.
- High confidence claim without strong/medium evidence fails.
- Source `allowed=false` cannot support verified claim.
- Contradicting claims can be represented and counted.
- Cascade impact for claim marked wrong downgrades dependent objects in report.
- Cascade impact for source marked unreliable affects linked claims.
